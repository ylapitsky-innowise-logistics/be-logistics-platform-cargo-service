

## Флоу загрузки товара с картинками в приложение:
### 🏛️ Паттерн: **Раздельные запросы** (Двухфазная загрузка со стороны фронтенда)
Это классический подход для большинства современных REST API и SPA/мобильных приложений.

Как выглядит флоу:
1. Фронтенд отправляет чистый JSON на `POST /api/v1/cargos`. Сервис сохраняет данные в _Postgres_, закрывает транзакцию и возвращает созданный cargoId.
2. Фронтенд получает cargoId и сразу же отправляет второй запрос (или серию запросов, если фоток много) на эндпоинт загрузки медиа: `POST /api/v1/cargos/{cargoId}/images` в формате `multipart/form-data`.

Плюсы:
- Идеально для транзакций: Транзакция в Postgres открывается на миллисекунды. База не ждет, пока тяжелые файлы запишутся по сети в Mongo.
- Удобный UX: Фронтенд может показывать пользователю красивый прогресс-бар загрузки каждой отдельной фотографии (10%... 50%... Готово).
- Отказоустойчивость: Если упадет сеть во время загрузки третьей картинки, тебе не нужно откатывать или пересоздавать сам товар в Postgres. Пользователь просто нажмет кнопку «повторить загрузку фото».

Минусы:
- В базе Postgres может временно появиться товар без картинок (хотя в логистике это нормальное состояние).

---
- Уровень изоляции Postgres: `READ COMMITTED`  
- `interface CargoRepository.findFirstNAvailableBySkuIdAndStatus`  использует пессимистическую блокировку, 
которая на уровне СУБД превращается в запрос SELECT ... FOR UPDATE.

> В режиме `READ COMMITTED` _Postgres_ обрабатывает этот замок следующим образом:  
> 1. Поток А пришел забронировать Cargo №1 и повесил на него замок.  
> 2. Поток Б пришел за тем же Cargo №1, наткнулся на замок и встал в очередь (уснул).  
> 3. Поток А успешно перевел статус Cargo №1 из AVAILABLE в BOOKED и сделал COMMIT.  
> 4. Поток Б просыпается. И вот тут включается магия READ COMMITTED в Postgres (механизм EvalPlanQual): 
он заново пересчитывает строку с учетом новых закомиченных данных потока А. 
Он видит, что статус у Cargo №1 больше не AVAILABLE, условие WHERE больше не выполняется, 
и он просто пропускает эту строку, уберегая тебя от дабл-букинга!
